# スプレッドシート連携とバックグラウンド送信機能の実装計画

売上データをGoogle Apps Script (GAS) 経由でスプレッドシートに送信し、オフライン時の再送や100件ごとのバッチ送信を行うための仕組みを実装します。

## 背景と目的
注文確定時に売上データをリアルタイムで送信しつつ、別途100件ごとに別のスプレッドシートにもバックアップとしてデータを送信します。通信環境が不安定な現場でもデータが欠損しないよう、ローカルストレージ（localStorage）を活用したキューイングと再送機構を構築します。

## User Review Required

> [!IMPORTANT]
> 以下の点について設計方針を確認してください。
> 1. **通し番号の仕様**: 通し番号はブラウザの `localStorage` で「1」から順にカウントアップする仕様でよろしいでしょうか？（※複数端末で同時に操作すると通し番号が重複する可能性がありますが、1つの端末での運用を想定した設計としています）
> 2. **保存上限の仕様**: ローカルに保存しておく履歴データの上限は、ひとまず「1000件」とし、1000件を超えたら古いものから削除する仕様で進めてよろしいでしょうか？

## Proposed Changes

### 1. `idex.html` (Webアプリ側)
注文確定の処理 (`confirmOrder`) をフックし、データ保存と送信ロジックを追加します。

- **データ構造の作成**: 
  A列: 通し番号, B列: 決済時刻, C〜H列: 商品1〜6の個数, I列: 合計個数, J列: 合計金額
- **ローカルストレージの活用**:
  - `sales_history`: 過去の全送信データを保存（最大1000件、超過で古い順に削除）
  - `sales_serial_number`: 通し番号のカウンター
  - `sync_queue_realtime`: リアルタイム送信待ちのキュー
  - `sync_queue_batch`: 100件バッチ送信待ちのキュー
- **バックグラウンド送信・再送ロジック**:
  - `setInterval` を用いて、1分ごとに未送信キュー (`sync_queue_realtime`, `sync_queue_batch`) をチェック。
  - キューがあれば `fetch` でGASの Web App URL（後から設定できるようにダミーURLを記載）にPOST送信。
  - 送信成功時はキューから削除。失敗時は次回の1分後のチェックで再送。
- **100件ごとのバッチ処理**:
  - 注文確定のたびに内部カウンターを増やし、100件に達したら、最新の100件のデータを配列として `sync_queue_batch` に追加し、送信トリガーをかける。

## Verification Plan

### Manual Verification
1. **通信成功のテスト**: 注文を確定し、ダミーのGASエンドポイントへ fetch が走ること（Console ログ）を確認。
2. **オフライン・再送のテスト**: ブラウザの Network タブで Offline に設定し注文を確定。キューに溜まることを確認。その後 Online に戻し、1分後の自動再送でキューが消化されることを確認。
3. **バッチ処理のテスト**: 内部のバッチカウンターを強制的に99に設定し、1件追加したタイミングで100件バッチ用の fetch が発火することを確認。
4. **上限動作のテスト**: 履歴の保存上限を一時的に10件などに設定し、超過時に古いデータが削除されることを確認。
